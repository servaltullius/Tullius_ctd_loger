# v0.2.42 — CTD/프리징 triage 재설계 + 덤프 intake/엔진 정확도 강화

## 핵심 변경
- EXE / system-victim CTD 해석을 전면 보강했습니다. 이제 `object-ref 중심 + 교차검증 게이트`를 사용해 후보를 정리하고, 약한 단서는 단독 승격하지 않도록 보수적으로 처리합니다.
- freeze / ILS 분석을 `deadlock_likely`, `loader_stall_likely`, `freeze_candidate`, `freeze_ambiguous` 상태로 분리했습니다. WCT, blackbox, snapshot, loading 문맥을 함께 사용합니다.
- recapture 정책을 `crash_richer`, `crash_full`, `freeze_snapshot_richer`로 세분화했습니다. 애매한 사고를 무조건 full dump로 올리지 않고, 부족한 신호에 맞춰 한 단계씩 강화합니다.
- 행동 우선 후보 이름을 `plugin filename > dll filename > mod folder name > friendly label` 순으로 고정했습니다. 모호한 표시명보다 실제로 찾을 수 있는 식별자가 먼저 보입니다.

## WinUI / 사용성
- 시작 화면을 `최근 발견된 덤프`와 `덤프 출력 위치` 중심으로 재구성했습니다. MO2 `overwrite` 또는 `OutputDir`을 기준으로 dump를 찾고, generic `CrashDumps`는 자동 탐색에서 제외했습니다.
- triage 읽기 순서를 `CrashLogger 기준 -> 근거 합의 -> 다음 조치`로 재배치했습니다. `Fault module`은 참고용 crash context로 내렸고, 권장 조치와 recapture context를 별도 설명 계층으로 분리했습니다.
- 후보 카드와 상단 요약은 친화 라벨보다 `ESP/ESM` 또는 `DLL` 파일명을 우선 표시합니다. `Faction Ranks` 같은 라벨이 실제 파일 식별자를 가리지 않도록 정리했습니다.
- `recapture_evaluation`을 WinUI recommendations 영역에 표시해, 왜 richer profile이나 snapshot 재수집이 걸렸는지 UI에서 바로 읽을 수 있게 했습니다.

## 엔진 변경
- `DumpMode` 단일 3단 구조를 incident-aware capture profile로 확장했습니다.
- freeze/manual capture에 opt-in `PSS snapshot` 경로를 추가했습니다.
  - 기본값은 여전히 `EnablePssSnapshotForFreeze=0`입니다.
- blackbox에 `module load/unload`, `thread create/exit`, `first-chance exception`을 추가했고, analyzer가 이를 loader stall / freeze / EXE CTD 해석에 소비합니다.
- symbol/runtime health 진단과 `first_chance_context`, `freeze_analysis`, `recapture_evaluation`을 output/report에 연결했습니다.
- helper/runtime false positive 방지를 강화했습니다.
  - 정상 종료 / weak crash에서는 artifact cleanup과 deferred viewer suppression이 유지됩니다.
  - synthetic helper runtime trigger 테스트를 추가해 게임 없이 CTD/freeze 발동 여부와 오작동 방지 경로를 검증할 수 있습니다.

  - `MainWindowViewModel`, `MainWindow.xaml.cs`

## 주의사항
- generic Windows `CrashDumps`는 자동 intake 대상이 아닙니다. 툴리우스 출력 위치(`OutputDir` / MO2 `overwrite`) 기준으로 동작합니다.
- `PSS snapshot` 기반 freeze capture는 아직 opt-in 기능입니다.
- `first-chance`, `blackbox`, `recapture_evaluation`은 보강 신호이며 단독으로 원인을 확정하지 않습니다.
- 수동 WinUI headless smoke가 필요하면 `.github/workflows/winui-headless-smoke.yml`을 직접 실행해야 합니다.
