# Dump Discovery UX Design

**Date:** 2026-03-13

**Goal:** 사용자가 덤프 파일 경로를 직접 알지 못해도, 덤프툴 첫 화면에서 최근 dump를 바로 발견하고 분석까지 이어지게 만든다.

## Context

현재 시작 경험은 `Browse dump`와 경로 입력에 무게가 실려 있다. 이 구조는 `정확한 .dmp 위치를 이미 아는 사용자`에게는 빠르지만, 일반 사용자는 첫 단계에서 막힌다.

특히 다음 문제가 반복된다.

- 사용자가 dump 파일이 어느 폴더에 생기는지 모른다.
- MO2 `overwrite`처럼 상위 위치만 알고, 실제 `.dmp` 최종 위치는 모르는 경우가 많다.
- 설치 구조가 모드팩마다 달라서 고정 경로 가이드만으로 해결되지 않는다.
- 자동 탐색이 실패했을 때도 다음 행동이 명확하지 않다.

## Decision

### 1. 시작 화면의 중심을 `최근 발견된 덤프`로 바꾼다

앱을 열면 가장 먼저 `최근 발견된 덤프` 목록을 보여준다.

각 항목은 최소한 아래 정보를 포함한다.

- 파일명
- 최근 수정 시각
- 파일 크기
- 발견 위치 라벨
- `Analyze` 액션

기존 `Browse dump`는 제거하지 않고 보조 액션으로 내린다.

### 2. 자동 스캔은 빠르고 좁게 시작한다

앱 시작 시 `빠른 자동 스캔`을 수행한다.

기본 검색 집합은 아래 세 종류다.

- `학습 경로`: 과거에 dump를 실제로 열거나 분석에 성공했던 위치
- `등록 경로`: 사용자가 추가한 dump 검색 위치
- `자동 경로`: 공통적으로 자주 쓰는 위치

검색 우선순위는 `학습 경로 > 등록 경로 > 자동 경로`로 둔다.

### 3. 사용자는 `정확한 dump 폴더`가 아니라 `덤프 검색 위치`를 등록한다

사용자-facing 용어는 `덤프 검색 위치`로 통일한다.

이유:

- 사용자는 정확한 dump 최종 폴더를 모르는 경우가 많다.
- MO2 `overwrite` 같은 상위 폴더만 알아도 충분해야 한다.
- 등록된 검색 위치는 하위 폴더까지 재귀 검색해야 실제 사용성이 올라간다.

즉, 사용자는 `정확한 .dmp 파일 생성 폴더`를 맞추는 대신, `이 근처에서 dump를 찾아라`를 알려주는 구조가 된다.

### 4. 못 찾았을 때는 직접적으로 안내한다

자동 스캔 결과가 없으면 empty state는 직접적인 문구를 사용한다.

- `덤프를 찾지 못했습니다`
- `MO2 overwrite 또는 사용자 지정 폴더를 추가하세요`
- `폴더 관리`
- `다시 스캔`
- `직접 선택`

완곡한 표현보다 다음 행동을 바로 지시하는 문구가 맞다.

### 5. `폴더 관리`는 첫 화면에서 바로 접근 가능해야 한다

`폴더 관리`는 설정 안에 숨기지 않고 첫 화면 보조 액션으로 둔다.

이 문제는 앱 사용 시작점에서 발생하므로, 진입점도 시작 화면에 있어야 한다.

설정 페이지에 같은 관리 UI를 두는 것은 허용하지만, 주 진입점은 첫 화면이다.

### 6. 알고 있는 위치의 새 dump만 자동 반영 대상으로 본다

새 dump 자동 반영은 `이미 알고 있는 경로`에서만 목표로 한다.

- 등록 경로
- 학습 경로
- 자동 경로

전혀 모르는 커스텀 위치의 새 dump를 완전 자동 발견하는 것은 목표에 넣지 않는다. 그 문제는 경로 탐지보다 넓은 설치 구조 추론 문제라서, 지금 단계에서는 비용 대비 효율이 낮다.

## Non-goals

- 분석 엔진 규칙 변경
- 결과 해석 화면 구조 개편
- 전체 디스크 스캔
- MO2/Vortex 모든 구조를 사전 하드코딩

## Acceptance Criteria

- 첫 화면의 주 흐름이 `파일 탐색기 열기`가 아니라 `최근 dump 선택`이 된다.
- 사용자는 `덤프 검색 위치`를 여러 개 등록할 수 있다.
- 등록 위치는 하위 폴더까지 검색된다.
- 검색 결과는 최근 수정 시각 기준 내림차순으로 정렬된다.
- 자동 탐색 실패 시 `MO2 overwrite 또는 사용자 지정 폴더를 추가하세요` 같은 직접적 안내가 보인다.
- `폴더 관리`는 첫 화면에서 바로 접근 가능하다.
